Listojo CI/CD Plan for Render and Railway

Date: April 21, 2026

Prepared for: Listojo

1. Executive Summary

Listojo can support a clean CI/CD pipeline on either Render or Railway. Based on the current repository structure, Render is the better first production target because it provides a simpler deployment path for a Django monolith and supports automatic deployment only after GitHub Actions checks pass.

Railway is also a valid option, especially if Listojo later expands into a multi-service architecture with workers, cron jobs, preview environments, or more infrastructure flexibility.

Recommendation: Deploy the current Django monolith to Render first, with GitHub Actions for CI and Render auto-deploy configured to run after CI checks pass.

2. Current State of the Application

The current repository is not fully production-ready for CI/CD deployment yet. The main issues identified are:

  • The app currently uses SQLite in Django settings.
  • There is no production WSGI app server dependency such as gunicorn.
  • There is no STATIC_ROOT configured for collected static files.
  • There is no production static-file serving setup such as WhiteNoise.
  • There is no deploy configuration file such as render.yaml, railway.toml, Dockerfile, or build.sh.
  • The app has not yet been wired for DATABASE_URL-based deployment.

3. Target CI/CD Model

CI Layer

CI should run on every pull request and every push to the main branch.

  • Install Python dependencies
  • Run Django migrations in CI database
  • Run python manage.py check --deploy
  • Run automated tests
  • Add linting and formatting later if needed

CD Layer

CD should only run after CI succeeds on the main branch.

  • Deploy the latest commit
  • Run database migrations before cutover
  • Collect static files
  • Start the application with gunicorn
  • Validate health through a health-check endpoint

4. Production Hardening Required Before Deployment

Dependencies to Add

  • gunicorn for production application serving
  • whitenoise for static-file serving
  • dj-database-url for environment-based database config
  • psycopg or psycopg[binary] for PostgreSQL connectivity

Django Settings Changes

  • Replace SQLite-only config with DATABASE_URL support
  • Add STATIC_ROOT = BASE_DIR / "staticfiles"
  • Add WhiteNoiseMiddleware immediately after SecurityMiddleware
  • Add SECURE_PROXY_SSL_HEADER
  • Ensure production-safe ALLOWED_HOSTS and CSRF_TRUSTED_ORIGINS
  • Add a lightweight /health endpoint returning HTTP 200
Important: Without these changes, the app may still build successfully on a platform, but fail at runtime, fail to serve static files correctly, or fail health checks.

5. Render Deployment Plan

Render is the recommended first platform for Listojo’s current architecture.

Recommended Render Service Setup

  • One Render Web Service for the Django application
  • One Render PostgreSQL database
  • GitHub repository linked to the service
  • Auto-deploy set to After CI Checks Pass

Render Build and Start Commands

Build command

pip install -r requirements.txt && python manage.py collectstatic --noinput

Pre-deploy command

python manage.py migrate --noinput

Start command

gunicorn classifieds_project.wsgi:application --bind 0.0.0.0:$PORT

Render Advantages for Listojo

  • Simple Django-friendly deployment workflow
  • Clean GitHub integration
  • Native support for deploying only after CI checks pass
  • Good fit for a modular monolith

6. Railway Deployment Plan

Railway is a good second option if Listojo later grows into a more service-oriented platform.

Recommended Railway Service Setup

  • One Railway service for the Django web app
  • One Railway PostgreSQL service
  • GitHub autodeploy enabled
  • Wait for CI enabled so deployment only proceeds after successful GitHub checks
  • Health check path configured to /health

Railway Build and Start Commands

Build command

pip install -r requirements.txt && python manage.py collectstatic --noinput

Pre-deploy command

python manage.py migrate --noinput

Start command

gunicorn classifieds_project.wsgi:application --bind 0.0.0.0:$PORT

Railway Advantages for Listojo

  • Better flexibility for future multi-service growth
  • Good support for workers, cron jobs, and preview environments
  • Works well if the platform later adds ingestion jobs or async workflows

7. Recommended GitHub Actions CI Workflow

The following GitHub Actions workflow is an appropriate starting point:

name: CI

on:
  pull_request:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_DB: app_test
          POSTGRES_USER: postgres
          POSTGRES_PASSWORD: postgres
        ports:
          - 5432:5432
        options: >-
          --health-cmd="pg_isready -U postgres -d app_test"
          --health-interval=10s
          --health-timeout=5s
          --health-retries=5

    env:
      DJANGO_SECRET_KEY: test-secret-key
      DJANGO_DEBUG: "false"
      DJANGO_ALLOWED_HOSTS: localhost,127.0.0.1,testserver
      DATABASE_URL: postgres://postgres:postgres@127.0.0.1:5432/app_test

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt

      - name: Run migrations
        run: python manage.py migrate --noinput

      - name: Run checks
        run: python manage.py check --deploy

      - name: Run tests
        run: python manage.py test

8. Required Environment Variables

The production deployment should use environment variables rather than local file defaults.

Variable Purpose
DJANGO_SECRET_KEY Django application secret
DJANGO_DEBUG Must be false in production
DJANGO_ALLOWED_HOSTS Allowed domains and platform URLs
DJANGO_CSRF_TRUSTED_ORIGINS Trusted HTTPS origins
DATABASE_URL PostgreSQL connection string
DEFAULT_FROM_EMAIL Outbound email identity
EMAIL_BACKEND Email transport backend
EMAIL_HOST SMTP host
EMAIL_PORT SMTP port
EMAIL_HOST_USER SMTP username
EMAIL_HOST_PASSWORD SMTP password
GOOGLE_MAPS_API_KEY Maps integration key

9. Phased Rollout Plan

Phase 1: Production Readiness

  • Update dependencies
  • Switch to PostgreSQL-ready settings
  • Add static-file production support
  • Add a health endpoint

Phase 2: Continuous Integration

  • Add GitHub Actions CI workflow
  • Run tests on every pull request and push
  • Run Django deployment checks

Phase 3: Continuous Deployment

  • Connect the repository to Render or Railway
  • Configure build, migration, and start commands
  • Gate deployment on successful CI

Phase 4: Operational Maturity

  • Add a staging environment
  • Move media storage off local filesystem if required
  • Add monitoring and alerting
  • Add error tracking such as Sentry
  • Document rollback and recovery runbooks

10. Final Recommendation

For Listojo’s current state as a Django monolith, the best immediate path is:

  • Prepare the app for production
  • Add GitHub Actions CI
  • Deploy to Render with auto-deploy only after CI passes

Railway should remain a strong future option if the platform grows into multiple services, background workers, ingestion pipelines, or preview-environment-heavy workflows.
